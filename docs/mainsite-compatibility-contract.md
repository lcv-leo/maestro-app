# MainSite Compatibility Contract

Status: compatibility baseline extracted from `admin-app` and `mainsite-app` on 2026-04-26.

Maestro must be able to read and write `bigdata_db.mainsite_posts` directly, while producing content that renders in `mainsite-app/mainsite-frontend/PostReader` exactly like content produced by `admin-app/MainSite/PostEditor`.

This is not an approximate Markdown export. The persisted `content` column must be MainSite-compatible HTML.

Because the required result is exact PostEditor behavior, Maestro's MainSite editor is a PostEditor parity module. A generic TipTap editor is not acceptable for this path.

## D1 Table Contract

Target database and table:

```text
bigdata_db.mainsite_posts
```

Current required fields used by admin and reader flows:

- `id INTEGER PRIMARY KEY AUTOINCREMENT`
- `title TEXT NOT NULL`
- `content TEXT NOT NULL`
- `author TEXT`
- `is_pinned INTEGER NOT NULL DEFAULT 0`
- `display_order INTEGER NOT NULL DEFAULT 0`
- `is_published INTEGER NOT NULL DEFAULT 1`
- `created_at DATETIME DEFAULT CURRENT_TIMESTAMP`
- `updated_at DATETIME`

Maestro writes must preserve MainSite behavior:

- New posts default to `is_pinned = 0`.
- New posts default to `display_order = 0` unless explicitly chosen.
- `is_published` is operator-controlled and defaults to draft-safe behavior in Maestro UI.
- Updates must set `updated_at = CURRENT_TIMESTAMP`.
- If Maestro later supports changing post IDs, it must preserve known foreign-key references the same way the admin route does.

## PostEditor Persistence Path

The current admin flow is:

1. TipTap document is rendered by `editor.getHTML()`.
2. Empty content is rejected when it is missing or equals `<p></p>`.
3. Before save, non-YouTube links are normalized with `target="_blank"` and `rel="noopener noreferrer"`.
4. The admin backend sanitizes content with a parser allowlist before writing to D1.
5. The MainSite frontend sanitizes again before rendering.

Maestro must implement the same pre-save and server-side sanitizer semantics before any direct D1 write.

Before save, Maestro must also run the Link Integrity Engine. Sanitized HTML is not enough: the URL must be syntactically safe, mechanically reachable or explicitly classified, and editorially capable of supporting its claim.

## HTML Structures Maestro Must Emit

Supported editorial blocks:

- Paragraphs and headings `h1` through `h6`.
- Strong, emphasis, underline, strike, mark, subscript, superscript, inline code.
- Blockquote, code block, horizontal rule.
- Ordered and unordered lists.
- Task lists using `ul data-type="taskList"` and `li data-type="taskItem" data-checked="..."`.
- Tables with `table`, `thead`, `tbody`, `tr`, `th`, `td`, `col`, `colgroup`, colspan and rowspan.
- Links with safe `href`, `target`, `rel`, and `title`.
- Mentions as `span class="editor-mention"`.
- Semantic images as `figure class="tiptap-figure"` with nested `img` and optional `figcaption`.
- Resizable standalone images with width data/style compatible with TipTap output.
- YouTube embeds as `div data-youtube-video` containing an `iframe`.

## Style Constraints

The admin/worker sanitizer currently permits only narrow inline style behavior for persisted content. The important stable style surface is:

- `text-align: left | right | center | justify`
- figure and image width generated by TipTap structures
- paragraph heading alignment and reader CSS handling

The current backend allowlist does not preserve every TipTap editing-time style. Maestro must prefer semantic HTML over arbitrary inline styling and must test each generated construct through the same sanitizer before D1 write.

## Renderer Expectations

`PostReader`:

- Detects HTML by checking for tags.
- Sanitizes with DOMPurify.
- Adds `iframe` support.
- Allows `target`, `rel`, `data-type`, `data-checked`, and `style`.
- Converts internal links for known LCV domains into relative paths.
- Restores external non-YouTube links to open in a new tab.
- Renders HTML inside `.html-content`.

The CSS then styles:

- `.post-reader .html-content p`
- `.post-reader .html-content table`
- `.post-reader figure.tiptap-figure`
- `.post-reader .editor-mention`
- `.post-reader .html-content ul[data-type="taskList"]`
- `.post-reader .html-content li[data-type="taskItem"]`

## Compatibility Tests Required Before Stable Release

Maestro needs golden fixtures that round-trip through:

1. Maestro generated HTML.
2. Maestro MainSite sanitizer.
3. D1 insert/update payload.
4. PostReader DOMPurify render.
5. Visual screenshot comparison against an equivalent PostEditor-authored post.

Minimum fixture set:

- Plain academic article with headings and paragraphs.
- ABNT-style references with links.
- Table-heavy article.
- Figure with caption.
- Standalone image with width.
- YouTube embed.
- Task list.
- Mention.
- Markdown import.
- Markdown plus HTML import.
- PDF-derived article.

No direct write to `mainsite_posts` should be exposed as a stable feature until these fixtures pass.

## Drift Checks

Before a stable release, CI should compare Maestro's `src/editor/posteditor/` compatibility module against the current `admin-app/MainSite/PostEditor` source or an explicit reviewed snapshot. Any drift must either be ported or documented as intentionally non-applicable.
